## S13-1: Inbound Payment Webhook + Auto-Match (F-IF-2)

**Monday.com:** 11909951085
**Priority:** P1 (High)
**Story points:** 13

### Problem

Debtor payment recording is currently 100% manual — an operator must click a button in the admin portal for each payment. At scale, this is the operational bottleneck for invoice factoring. The platform needs an inbound webhook endpoint that payment providers (bank transfers, mobile money) can call when a debtor payment lands, with automatic matching to the correct invoice.

### What to Build

#### 1. Webhook Endpoint

Create a new controller in `apps/rest-server/src/`:

**File:** `apps/rest-server/src/debtor-payment-webhook/debtor-payment-webhook.controller.ts`

Follow the exact pattern of `wallet-webhook.controller.ts`:
- `@Public()` — bypasses JWT AuthGuard (authenticates via HMAC)
- `@Controller('webhooks')` routing
- HMAC signature validation using `x-signature` header against `WEBHOOK_SECRET_{PROVIDER}` env var
- Returns `202 Accepted` immediately; actual processing is async

**Endpoint:** `POST /webhooks/{provider}/debtor-payment`

**Payload shape:**
```typescript
interface DebtorPaymentWebhookPayload {
  /** Provider's own transaction reference */
  transactionRef: string;
  /** Amount received in the payment */
  amount: string;
  /** ISO-4217 currency code */
  currency: string;
  /** Identifying info for matching — at least one must be present */
  invoiceNumber?: string;
  debtorRef?: string;
  paymentRef?: string;
  /** Provider-specific metadata */
  metadata?: Record<string, unknown>;
}
```

**Create DTOs:** `debtor-payment-webhook.dto.ts` with class-validator decorators. `amount` must be a valid Decimal string. At least one of `invoiceNumber | debtorRef | paymentRef` must be present (custom validator).

**Create module:** `debtor-payment-webhook.module.ts` — register in `app.module.ts`.

#### 2. Matching Service

**File:** `services/process-engine/src/factoring/debtor-payment-matching.service.ts`

The matching logic should attempt to match an inbound payment to an invoice using a waterfall strategy:

```
1. Exact match by invoiceNumber + currency + tenantId
   (invoiceNumber is unique per tenant+seller, so filter by debtor_notified or payment_received status)
2. If no invoiceNumber: match by debtorRef → look up debtor, then find
   their oldest outstanding invoice in debtor_notified status (FIFO)
3. If no match found: emit DEBTOR_PAYMENT_UNMATCHED event and return
   { matched: false, reason: 'no_matching_invoice' }
```

When matched:
- Call `reserveService.recordDebtorPayment(tenantId, invoiceId, { amount, paymentRef: transactionRef, idempotencyKey: transactionRef })`
- The existing `recordDebtorPayment` already handles full/partial payments, status transitions, reserve release triggers, and debtor risk reassessment — **do not duplicate** that logic
- Emit `DEBTOR_PAYMENT_MATCHED` event with `{ invoiceId, amount, transactionRef, matchStrategy: 'invoice_number' | 'debtor_ref' | 'fifo' }`

When unmatched:
- Emit `DEBTOR_PAYMENT_UNMATCHED` event with `{ transactionRef, amount, currency, reason }`
- Log at `warn` level for operator visibility
- The admin portal already has the manual `recordDebtorPayment` button as fallback

**Important:** The matching service must be idempotent on `transactionRef`. Use `transactionRef` as the `idempotencyKey` passed to `recordDebtorPayment`. The existing idempotency check in `recordDebtorPayment` (line 108 of `reserve.service.ts`) will handle duplicate webhooks.

#### 3. Event Contracts

Add to `packages/event-contracts/`:
- `EventType.DEBTOR_PAYMENT_MATCHED`
- `EventType.DEBTOR_PAYMENT_UNMATCHED`

Follow existing patterns in the event-contracts package.

#### 4. Tests

- **Unit tests** for matching service: exact match, FIFO match, no match, duplicate webhook (idempotency), currency mismatch
- **Unit tests** for webhook controller: HMAC validation, malformed payload, missing required fields, happy path
- **Integration test:** Add a webhook-to-settlement scenario to `factoring-lifecycle.integration.spec.ts` — submit invoice → generate offer → accept → fund → debtor payment webhook → verify auto-match → verify reserve release

#### 5. Admin Portal — Webhook Activity Log

On the invoice detail slide-over panel, add a small section (below the existing payment info) showing webhook-received events for that invoice. This is read-only and draws from the audit log / event history. If no webhook events exist for the invoice, don't show the section.

### Verification

```bash

pnpm --filter process-engine test -- --testPathPattern debtor-payment-matching
pnpm --filter process-engine test -- --testPathPattern factoring-lifecycle
```

## S13-2: `debtorPaidAt` Field for Accurate Risk Scoring (F-IF-3)

**Monday.com:** 11910003633
**Priority:** P2 (Medium)
**Story points:** 3

### Problem

`debtor.service.ts` line 469–473 uses `updatedAt` as a proxy for the actual payment date when calculating debtor risk scores. `updatedAt` advances on **any** mutation (e.g., adding a verification note, changing metadata), making the payment-delay calculation unreliable. This affects `averagePaymentDays` and the `paymentHistoryFactor` in the debtor risk score, which in turn affects advance rate calculations for future invoices.

### What to Build

#### 1. Schema Migration

Add to `Invoice` model in `packages/database/prisma/schema.prisma`:

```prisma
debtorPaidAt       DateTime?           @map("debtor_paid_at") @db.Timestamptz(6)
```

Place it after `debtorPaymentRef` for logical grouping.

**Migration file:** `packages/database/prisma/migrations/20260508000000_add_invoice_debtor_paid_at/migration.sql`

```sql
-- S13-2: Track actual debtor payment date for accurate risk scoring.
-- Replaces the unreliable updatedAt proxy in debtor risk calculation.
ALTER TABLE "invoices" ADD COLUMN "debtor_paid_at" TIMESTAMPTZ(6);
```

Run `npx prisma generate` after.

#### 2. Set `debtorPaidAt` on First Payment

In `services/process-engine/src/factoring/reserve.service.ts`, inside `recordDebtorPayment`:

After the payment is validated and before the invoice update (around line 180), add logic to set `debtorPaidAt` **only on the first payment event** (when the existing `amountReceived` is `0` or `null`):

```typescript
// Set debtorPaidAt on the first payment event — used by risk scoring
// to calculate actual payment delay instead of the unreliable updatedAt.
const isFirstPayment = !invoice.amountReceived ||
  new Decimal(String(invoice.amountReceived)).isZero();
```

Then in the `prisma.invoice.update` call (around line 183), add:

```typescript
data: {
  // ... existing fields ...
  ...(isFirstPayment ? { debtorPaidAt: new Date() } : {}),
}
```

#### 3. Update Risk Scoring

In `services/process-engine/src/factoring/debtor.service.ts`, `assessRisk` method:

**Line 450–454** — Add `debtorPaidAt` to the `select` clause:
```typescript
select: {
  id: true,
  status: true,
  dueDate: true,
  debtorPaidAt: true,  // ← add
  updatedAt: true,     // keep as fallback
},
```

**Lines 469–473** — Replace the payment date logic:
```typescript
// Use debtorPaidAt (set on first payment) when available;
// fall back to updatedAt for invoices paid before this field existed.
const actualPaymentDate = inv.debtorPaidAt ?? inv.updatedAt;
const daysLate = daysBetweenUtc(inv.dueDate, actualPaymentDate);
```

Remove or update the `v1 LIMITATION` comment to reflect the new field.

#### 4. GraphQL Type

Add to `InvoiceType` in `apps/graphql-server/src/graphql/types/factoring.type.ts`:

```typescript
@Field({ nullable: true }) debtorPaidAt?: Date;
```

Place it after the `debtorPaymentRef` field (line 137).

#### 5. Tests

- **Unit test** in `reserve.service.spec.ts`: verify `debtorPaidAt` is set on first payment, not on subsequent partial payments
- **Unit test** in `debtor.service.spec.ts`: verify `assessRisk` prefers `debtorPaidAt` over `updatedAt` when both are present; falls back to `updatedAt` when `debtorPaidAt` is null

## S13-3: Grace-Period Expiry Scheduler Job (F-IF-4)

**Monday.com:** 11909960228
**Priority:** P2 (Medium)
**Story points:** 5

### Problem

`RecourseService.handleWithRecourse` correctly sets `recourseGraceEndAt` on the invoice metadata when a with-recourse invoice defaults (line 244 of `recourse.service.ts`). `enforceGracePeriodElapsed` correctly validates and processes grace expiry (line 136). But **nothing calls `enforceGracePeriodElapsed`** — there is no scheduler job that scans for invoices past their grace end date. This means with-recourse defaults enter a dead zone where the grace period silently elapses but enforcement never triggers.

### What to Build

#### 1. Scheduler Job

**File:** `apps/scheduler/src/jobs/recourse-grace-expiry.job.ts`

Follow the exact pattern of `invoice-offer-expiry.job.ts` (per-tenant fan-out, per-tenant error isolation, `enterTenantContext`):

```typescript
@Injectable()
export class RecourseGraceExpiryJob {
  private readonly logger = new Logger('RecourseGraceExpiryJob');

  constructor(
    private readonly prisma: PrismaService,
    private readonly recourseService: RecourseService,
  ) {}

  /** Daily at 07:00 UTC — after aging (06:00) has classified new defaults. */
  @Cron('0 7 * * *')
  async handleCron(): Promise<void> {
    // 1. Get all active tenants (platform-admin context)
    // 2. For each tenant:
    //    a. Find invoices where:
    //       - status = 'defaulted'
    //       - recourseType = 'with_recourse'
    //       - metadata->>'recourseGraceEndAt' IS NOT NULL
    //       - metadata->>'recourseGraceEndAt' <= now()
    //       - metadata->>'recourseEnforcedAt' IS NULL  (not already enforced)
    //    b. For each match, call recourseService.enforceGracePeriodElapsed(tenantId, invoiceId)
    //    c. try/catch per invoice so one failure doesn't block others
  }
}
```

**Important query note:** `recourseGraceEndAt` is stored in the JSONB `metadata` field, not as a top-level column. Use Prisma's JSON filtering:

```typescript
this.prisma.invoice.findMany({
  where: {
    tenantId: tenant.id,
    status: InvoiceStatus.defaulted,
    recourseType: RecourseType.with_recourse,
    metadata: {
      path: ['recourseGraceEndAt'],
      not: Prisma.DbNull,
    },
  },
})
```

Then filter in application code for `graceEndAt <= now()` and `!metadata.recourseEnforcedAt` (Prisma JSON filtering doesn't support date comparisons on string values within JSONB).

After `enforceGracePeriodElapsed` succeeds, stamp `metadata.recourseEnforcedAt = new Date().toISOString()` to prevent re-processing on the next scan. This should be done inside `enforceGracePeriodElapsed` itself — add a metadata update at the end of the method (after line 210 in `recourse.service.ts`), merging `recourseEnforcedAt` into existing metadata.

#### 2. Register in Scheduler Module

Add `RecourseGraceExpiryJob` to `apps/scheduler/src/scheduler.module.ts`:
- Import `RecourseService` from `@lons/process-engine`
- Add both to the `providers` array

#### 3. Tests

**File:** `apps/scheduler/src/jobs/recourse-grace-expiry.job.spec.ts`

Follow the pattern of `invoice-offer-expiry.job.spec.ts` (same `enterTenantContext` mock):

- Finds and enforces grace-expired invoices across tenants
- Skips invoices where grace hasn't elapsed yet
- Skips invoices already enforced (`recourseEnforcedAt` present)
- Continues processing other tenants/invoices if one fails
- Handles no expired invoices gracefully

Also add a unit test in `recourse.service.spec.ts` verifying that `enforceGracePeriodElapsed` now stamps `recourseEnforcedAt` on the metadata.

### Verification

```bash
pnpm --filter scheduler test -- --testPathPattern recourse-grace-expiry
pnpm --filter process-engine test -- --testPathPattern recourse.service
```

## S13-4: Debtor/Seller Nested Resolvers on InvoiceType (F-IF-7)

**Monday.com:** 11909953391
**Priority:** P3 (Low)
**Story points:** 2

### Problem

When querying invoices via GraphQL, the `sellerId` and `debtorId` fields return raw UUIDs. The `FactoringResolver` (line 76 of `factoring.resolver.ts`) is declared as `@Resolver(() => InvoiceType)` but has no `@ResolveField` decorators for the `seller` and `debtor` relations. Operators in the admin portal see truncated UUIDs instead of company/customer names.

### What to Build

#### 1. Add Fields to InvoiceType

In `apps/graphql-server/src/graphql/types/factoring.type.ts`, add to `InvoiceType`:

```typescript
/** Resolved via @ResolveField in FactoringResolver. */
@Field(() => DebtorType, { nullable: true })
debtor?: DebtorType;

/** Resolved via @ResolveField in FactoringResolver — the seller is a Customer. */
@Field(() => CustomerType, { nullable: true })
seller?: CustomerType;
```

Import `CustomerType` from the appropriate types file. These fields are nullable because the resolver loads them on demand.

#### 2. Add ResolveField Decorators

In `apps/graphql-server/src/graphql/resolvers/factoring.resolver.ts`, add two `@ResolveField` methods:

```typescript
@ResolveField(() => DebtorType, { nullable: true })
async debtor(@Parent() invoice: InvoiceType): Promise<DebtorType | null> {
  if (!invoice.debtorId) return null;
  const debtor = await this.prisma.debtor.findFirst({
    where: { id: invoice.debtorId, deletedAt: null },
  });
  return debtor as unknown as DebtorType | null;
}

@ResolveField(() => CustomerType, { nullable: true })
async seller(@Parent() invoice: InvoiceType): Promise<CustomerType | null> {
  if (!invoice.sellerId) return null;
  const customer = await this.prisma.customer.findFirst({
    where: { id: invoice.sellerId, deletedAt: null },
  });
  return customer as unknown as CustomerType | null;
}
```

Add `import { Parent, ResolveField } from '@nestjs/graphql';` and import `CustomerType`.

**N+1 note:** These resolvers will fire per invoice in a list query. This is acceptable for v1 because invoice lists are capped at 20 items per page (cursor pagination). A DataLoader optimization can be added in a future sprint if profiling shows it's needed.

#### 3. Admin Portal — Display Names

In the admin portal invoice list and detail views, update the display to show:
- **Debtor column:** `invoice.debtor?.companyName ?? invoice.debtorId` (fall back to ID if resolver returns null)
- **Seller column:** `invoice.seller?.fullName ?? invoice.seller?.companyName ?? invoice.sellerId`

Update the GraphQL queries in the admin portal factoring pages to request `debtor { companyName }` and `seller { fullName }` nested fields.

#### 4. Tests

Add tests to `factoring.resolver.spec.ts`:
- `debtor` resolve field returns debtor object when debtorId exists
- `seller` resolve field returns customer object when sellerId exists
- Both return null for missing IDs

### Verification

```bash
pnpm --filter graphql-server test -- --testPathPattern factoring.resolver
pnpm --filter admin-portal build
```

## S13-5: BNPL Early Settlement Discount — Use Remaining Balance (F-IF-8)

**Monday.com:** 11909951133
**Priority:** P2 (Medium)
**Story points:** 3

### Problem

In `services/process-engine/src/bnpl/bnpl-installment.service.ts` lines 781–785, `totalRemaining` sums the full `inst.amount` for pending installments, ignoring any `paidAmount` already applied via partial payments:

```typescript
// CURRENT (BUG):
let totalRemaining = '0';
for (const inst of pendingInstallments) {
  totalRemaining = add(totalRemaining, String(inst.amount));
}
```

If a customer has partially paid an installment (e.g., `amount=1000, paidAmount=400`), the discount is calculated on the full `1000`, not the remaining `600`. The customer gets a larger discount than warranted, and the books don't balance.

### What to Fix

#### 1. Fix `totalRemaining` Calculation

Replace lines 781–785 with:

```typescript
// Total remaining = sum of (amount - paidAmount) for each pending installment.
// paidAmount tracks partial payments already applied to each installment.
let totalRemaining = '0';
for (const inst of pendingInstallments) {
  const instRemaining = subtract(String(inst.amount), String(inst.paidAmount ?? 0));
  totalRemaining = add(totalRemaining, instRemaining);
}
```

All math functions (`subtract`, `add`) are already imported from `@lons/common`.

#### 2. Fix Installment Closure

After the settlement payment is applied, each pending installment should be marked as paid. Verify that the closure logic (further down in the same method) sets `paidAmount = amount` (i.e., the full installment amount, not `totalRemaining / count`). The existing code should already do this correctly, but verify — if it uses the discounted amount instead of the full amount, that's a secondary bug.

#### 3. Tests

In the BNPL installment service spec file, add/update tests:

- **Partial payment scenario:** Create a transaction with 3 installments where installment #1 is fully paid, installment #2 has `paidAmount = 400` of `amount = 1000`, installment #3 is unpaid (`paidAmount = 0`, `amount = 1000`). Early settle. Verify `totalRemaining = 600 + 1000 = 1600` (not `1000 + 1000 = 2000`).
- **No partial payments scenario:** All pending installments have `paidAmount = 0`. Verify behavior is unchanged from current.
- **All partially paid:** Multiple installments with varying `paidAmount`. Verify discount applies to actual remaining balance.

```bash

```
